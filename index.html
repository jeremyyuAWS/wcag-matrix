How to read this version

  • Every assess/remediate cell in the table still shows ACP's shipped status (Certified/Guided/Human/N-A · Automatic/AI proposal/Manual/None) — that view didn't go away, and it's still code-verified against the July-14 infographic (see prior corrections, now folded into each drawer's §5).
  • What's new is the drawer. Click any cell and the top of the drawer (§§1–4) answers "what does this rule inherently allow," derived from WCAG's own requirement plus the file format's structural properties — not from ACP's roadmap. That classification is stable; it doesn't move when ACP ships more.
  • §5 "Where ACP is today" is the only part of the drawer that changes over time — and the two tiles above the table (ceiling vs ACP at ceiling) turn that into one honest headline: of the cells where a ceiling above N/A exists, how many has ACP actually reached.
  • N/A got cleaner under this lens: it's now a property of the rule+format pair (e.g. reflow presupposes a resizable viewport; paginated OOXML has none) rather than a placeholder for "not built yet." A few cells that read Human under the old build-status framing (1.4.4, 1.4.12, several NA-by-content-model rows) are reclassified N/A here because no ceiling above N/A exists for that format — not because nothing's shipped.
Ceiling claims (drawer §§2–4, table tiles) are argued from the WCAG 2.2 normative text and each format's structural model — see docs/deva-reconciliation.md and docs/tier-methodology.md for the reasoning basis. ACP-status claims (drawer §5, table cell colors) cite engine/office-analysers + vendored worker-python + api/ocr.py · textchecks.py · office_structure.py (detection) and api/remediate_office.py · remediate_pdf.py + HITL lane (api/ai.py) (remediation). rule-catalog.json/docs/rules = intent & spec. digital-accessibility/ (partner remediators) is gitignored and not deployed — never capability.

What Deva's checklist does and doesn't cover

From V3's "Gaps vs WCAG" sheet plus V4's applicability matrix (the newer, cell-level source the toggle above now uses) — the checklist and the 20 WCAG criteria this matrix tracks don't cover the same ground. Neither list is wrong; they answer different questions.

  • 5 items have no WCAG SC behind them at all — Accessible Fonts (sans-serif, 11pt: no success criterion mandates a typeface or size, a legitimate house-style check, not a conformance one) and, in XLSX, Name the Table / Named Ranges / Cell A1 Text / Unique Sheet Names (Microsoft authoring hygiene that helps screen-reader navigation, but maps to no criterion). None of these five appear as rows in the table above — reporting them as WCAG failures would misrepresent them.
  • 2 items check real WCAG criteria outside this matrix's 20-SC scope — Captioning for Multimedia (1.2.x, time-based media) and No Auto-Playing Audio (1.4.2, Audio Control). Both are real, checkable criteria; they're just not among the 20 this matrix was scoped to track.
  • 3 items are sub-checks, not separate criteria — Table Captions, Decorative Marking, and Uses Slide Layouts are useful operational decompositions of 1.3.1 and 1.1.1, not criteria of their own. They appear in the table under their parent SC's row, not as independent entries.
  • 8 WCAG criteria have no checklist item at all, per V4 — 1.3.3, 1.4.1, 1.4.11, 3.1.2, 2.1.2 (never had one) plus 1.4.4, 1.4.10, 1.4.12 (V3's own commentary called these "correctly absent" — viewport behaviors static documents can't fail — and V4's applicability matrix confirms they were never actual checklist items either). None of these eight appear as rows when the toggle above is on, and the drawer's "vs. Deva's checklist" comparison (§3) has no entry for them — there's nothing in her checklist to agree, augment, or disagree with.
  • V4 is more precise than V3 at cell level, not just SC level — several of the 12 SCs V4 does cover only apply to SOME formats (1.3.2/2.1.1/2.4.3: PPTX+PDF only; 1.4.5: DOCX+PDF only; 2.4.6/3.1.1: no XLSX item; 4.1.2: DOCX only conditionally, PDF only). Cells outside V4's applicability for a shown row dim rather than disappear — real ACP status is never hidden, only de-emphasized. V4's own file reports "34 applicable cells," but a direct count of every marked cell (and its own per-format column totals) both total 36 — a discrepancy in Deva's source file, noted here rather than silently corrected.
