# wcag-matrix — project context for Claude Code

## What this is
Interactive WCAG 2.1 AA assessment & remediation matrix for document formats (DOCX/XLSX/PPTX/PDF), built for the ACP (Accessibility Compliance Platform) GTM conversation with Deva. Single self-contained `index.html` (no build step) + methodology docs in `docs/`. Live at https://fabulous-crisp-14e424.netlify.app/ (Netlify Drop; redeploy by drag or `netlify deploy --prod --dir .`).

## Ground rules (non-negotiable)
1. **Every capability claim must be code-backed** against the ACP repo at `~/projects/acp` — a tier is claimed only if the detector/fixer is shipped and in the scan path. Catalog labels alone don't count.
2. **Determinism bar:** LLM in a decision path caps a rule at Guided/AI-Proposal. Certified/Automatic = same input → same output.
3. **Honest-partial rule:** detectors with common blind spots (e.g. XLSX contrast skips theme colors) are Guided, never Certified.
4. Known **catalog ≠ code drift** (disclose, don't hide): PPTX reading-order & contrast fixers implemented but catalog says human-only; DOCX/PDF bookmark + XLSX sheet-name fixers cataloged `auto` but NOT built; XLSX hidden-content auto-unhide implemented but cataloged human-only.

## index.html data model
All content lives in one `<script>` block:
- `ROWS` — 20 SC rows; per row: `a`/`r` = ACP's shipped assessment/remediation tier per format (C/Q/H/NA · A/AC/AP/AI/M/N/NA) — drives the table's cell colors/labels, the §2 "Today" chip (assessment), and the §4 "Current" card (remediation, via `RTIER_HOW`). `ae`/`re` = per-format mechanism detail (`b` argument bullets, `c` concessions, `x` counterargument, `ev` evidence paths, `cat` drift note) — bullets + evidence render in drawer §6 (collapsed "Technical notes"); concessions/counterargument/drift render in §5 (FAQ) instead.
- `RUBRIC` — **the rule-inherent ceiling** ("potential"), independent of ACP's shipped status: per SC, `plain` (2-sentence rule statement) + per-format `{aClass, aWhy, fClass, fWhy}`. `aClass` ∈ DET/MA/HUM/NAx, `fClass` ∈ MECH/GEN/REAUTH/NAx — both display through the **same three-way vocabulary** now: `RCLASS`/`FCLASS` render DET/MECH→"Rules-based", MA/GEN→"AI-assisted", HUM/REAUTH→"Human-assisted", NAx→"Not applicable". This is the taxonomy that should NOT churn as ACP ships more. Grounded in `docs/deva-reconciliation.md` (assess ceiling reasoning) and `docs/tier-methodology.md` (fix ceiling reasoning, reframed without the "shipped" qualifier).
- `RTIER_HOW` — maps ACP's shipped remediation tier (`ROWS.r[f]`, e.g. A/AC/AP/AI/M/N) onto the *same* DET/MA/HUM/NAx bucket as RUBRIC's `fClass`, so "current" and "potential" read on one shared scale in drawer §4. Verified per-tier against actual mechanism text, not assumed: AC (1.1.1's vision-conditional auto-apply) → AI-assisted; AP (1.3.2 xlsx hidden-content unhide, 2.4.2's filename-derived title) → Rules-based, because those specific AP cases are mechanical, not generative. **If a new SC ever uses AC/AP, re-check its actual mechanism before trusting this mapping** — it's not a generic rule, it's been verified case-by-case.
- `RANK` + the `.scopenote` in `remediationCVP()` — Rules-based/AI-assisted/Human-assisted is **not a strict quality ladder**; it only orders mechanism reliability *within a fixed scope*. Whenever ACP's current fix mechanically solves a narrower slice of a rule than the rule's ceiling covers (e.g. 1.3.2 xlsx: mechanically unhides content, but has no safe fix for merged cells, whose ceiling stays Human-assisted), Current can rank higher than Potential without that being a contradiction — it means ACP already nailed the easy sub-case, not that ACP exceeds the ceiling. `RANK` detects this (`RANK[curHow] > RANK[potClass]`) and auto-appends an explanatory note so the badges don't read as self-contradictory. As of this writing it fires on 7 cells: 1.3.2/xlsx, 2.4.6/docx+pptx, 2.4.2/docx+xlsx+pptx+pdf — all "presence-only mechanical fix vs. descriptive/generative ceiling" cases. If you add a new SC/format where this pattern could recur, don't hand-flag it — the check is generic and will catch it automatically as long as `fClass`/`RTIER_HOW` are set correctly.
- `DEVA` — per-SC: Deva's verdict verbatim (`v`, `t`) + our conclusion (`k`: agree|augment|disagree, `ours`, `note`); rendered as an aside under drawer §3. Also surfaces directly on the table: `DEVA_MARK` (in the table-build loop) tags each ASSESS cell (never remediate — her checklist has no remediation column) with † (augment) or ‡ (disagree), but **only where `RUBRIC[sc].formats[f].aClass !== "NAx"`** — a per-SC `k` sometimes covers only some formats (1.3.2's "disagree" is about XLSX/PPTX/PDF beating her class; DOCX is N/A and has no claim to compare, so it gets no mark even though the row's `k` is "disagree"). If you add a new DEVA entry, don't assume the mark should appear on all 4 assess cells — check whether the `note` text's claim actually applies format-by-format first.
- `W3SLUG` — per-SC anchor into https://www.w3.org/TR/WCAG22/ (canonical), W3C Understanding, and W3C Techniques (quickref) links — all three shown in drawer.
- `PRINCIPLE` — per-SC WCAG Principle→Guideline breadcrumb text (factual hierarchy, not the SC text itself) — shown under the h2.
- `QUOTE` — per-SC short (<15-word) excerpt of the SC's own normative wording, quotation-marked and attributed with a link back to the canonical text — NOT the full SC text (deliberately trimmed; don't expand these into full reproductions).
- `UPGRADE` — per-SC, one hand-authored sentence: the cheapest concrete build that would close the gap between current and potential. Shown directly in drawer §4 ("Gap to close") — not in the FAQ (moved there to avoid duplicating the same sentence twice in one drawer).

Drawer render: `renderDrawer(sc, f, trigger)` (called from both the table's click handler and cross-link clicks) builds a **unified 6-section drawer per (SC, format)**:
1. The rule, in plain words
2. Can this be assessed automatically? — RUBRIC ceiling badge + an inline "Today: …" chip (`assessTodayChip()`, from `ROWS.a[f]`) — assessment does NOT get the current/potential merge that remediation does; it's one badge with a status chip beside it.
3. Why it lands there [+ the Deva aside, if this SC has one]
4. **Remediation — current vs. potential** (`remediationCVP()`) — a side-by-side comparison card, not two separate sections: Current (ACP today, from `ROWS.r[f]` via `RTIER_HOW`) vs. Potential (RUBRIC's `fClass`), both labeled on the Rules-based/AI-assisted/Human-assisted scale, followed by `fWhy` and the `UPGRADE` "Gap to close" note. This replaced the old split of "§4 ceiling badge" + "§5 ACP-today status line" — don't reintroduce that split without a reason; the merge was a deliberate scope decision (assessment kept its old two-part structure, remediation didn't).
5. FAQ (`buildFAQ()`) — mostly derived from data already on `ROWS.ae`/`ROWS.re` (first argument bullet, concessions, counterargument, catalog drift), so it doesn't require separate per-format authoring. Does NOT repeat the "gap to close" content — that lives only in §4 now.
6. Technical notes (collapsed `<details>`) — mechanism bullets + evidence only; concessions/counterarguments/drift live in the FAQ instead, not duplicated here.

`linkify()` auto-links bare SC-code mentions (e.g. "1.3.2") inside plain/why/FAQ/gap text to that SC's own drawer — don't hand-author cross-link markup, it's automatic from any text that mentions another SC's code. Ceiling tiles above the table (`#ceilingTiles`, `#acpTiles`) are computed at load time by iterating `ROWS`×`RUBRIC` — never hand-count these; the IIFE right after the table-build loop is the single source.

**Accessibility of the tool itself** (dogfooded, since it's a WCAG demo): table cells are real `<button>` elements (not click-handler `<td>`s) with descriptive `aria-label`s — natively keyboard-reachable. The drawer (`#panel`) has `role="dialog" aria-modal="true" aria-labelledby="panel-title"`, a focus trap (Tab cycles within it), Escape-to-close, and focus returns to the triggering button on close (`lastTrigger` tracks this — always pass `trigger` to `renderDrawer()`, including from cross-link clicks, or focus-return breaks). If you add new interactive elements to the drawer template, they need `:focus-visible` styling (already covers `button`/`a`/`.xlink` generically) and must be reachable within the existing focus-trap query (`button, a[href], [tabindex="0"]`).

Validate any index.html edit: extract the script block and eval ROWS/RUBRIC/DEVA/W3SLUG/PRINCIPLE/QUOTE/UPGRADE/RTIER_HOW — every SC in ROWS must have a RUBRIC entry with all 4 formats populated, NA-consistency should hold (aClass NAx ⇒ fClass NAx), `buildFAQ()` should return ≥1 item for every SC×format, and `remediationCVP()`/`assessTodayChip()` should never render the string "undefined". A quick in-browser integrity check (paste into devtools or `javascript_tool`):
```js
for (const row of ROWS) { const r = RUBRIC[row.sc]; if (!r) console.error(row.sc, 'missing'); for (const f of ["docx","xlsx","pptx","pdf"]) { const e = r.formats[f]; if (!e || !e.aClass || !e.fClass) console.error(row.sc, f, 'incomplete'); if (buildFAQ(row, f, e).length === 0) console.error(row.sc, f, 'no FAQ'); if (remediationCVP(row, f, e).includes('undefined')) console.error(row.sc, f, 'cvp broken'); } }
```

## Mobile / responsive
- `#panel` goes full-width (`width:100%`, no left border) below 640px; `.cvp` (the current-vs-potential cards in §4) stacks vertically below the same breakpoint — both via one `@media (max-width:640px)` block. `#panel{overflow-x:hidden}` is a defensive second rule (deliberately separate, later in source order, so it overrides only the `overflow-x` half of the base rule's `overflow:auto` shorthand) — keep it if you ever add wide content (tables, code blocks) to the drawer.
- `#backdrop` (a fixed, semi-transparent overlay, click-to-close) plus `document.querySelector(".wrap").inert = true/false` toggle in `renderDrawer()`/`closePanel()` give the dialog real modal isolation — the background table is genuinely unclickable and untabbable while the drawer is open, not just visually dimmed. `aria-modal="true"` alone does NOT enforce this in browsers; `inert` is what actually does.
- **Known tooling gap, not a page bug:** this repo's dev-session browser-preview tool has a reproducible viewport desync at narrow widths — `window.innerWidth` reports a stale wide value (differs from `outerWidth`/`screen.width`) after `resize_window` to a mobile preset, and screenshots taken in that state show clipped/scrolled-looking text even though every computed style and `scrollLeft` checks out correct. Verify mobile CSS via `getComputedStyle()`/`matchMedia()` assertions (as done for this change), not by trusting narrow-viewport screenshot pixels in this environment — cross-check against a real device or a different browser tool if a pixel-level check is ever needed.
