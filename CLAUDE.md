# wcag-matrix ‚Äî project context for Claude Code

## What this is
Interactive WCAG 2.1 AA assessment & remediation matrix for document formats (DOCX/XLSX/PPTX/PDF), built for the ACP (Accessibility Compliance Platform) GTM conversation with Deva. Single self-contained `index.html` (no build step) + methodology docs in `docs/`. Live at https://fabulous-crisp-14e424.netlify.app/ (Netlify Drop; redeploy by drag or `netlify deploy --prod --dir .`).

## Ground rules (non-negotiable)
1. **Every capability claim must be code-backed** against the ACP repo at `~/projects/acp` ‚Äî a tier is claimed only if the detector/fixer is shipped and in the scan path. Catalog labels alone don't count.
2. **Determinism bar:** LLM in a decision path caps a rule at Guided/AI-Proposal. Certified/Automatic = same input ‚Üí same output.
3. **Honest-partial rule:** detectors with common blind spots (e.g. XLSX contrast skips theme colors) are Guided, never Certified.
4. Known **catalog ‚âÝ code drift** (disclose, don't hide): PPTX reading-order & contrast fixers implemented but catalog says human-only; DOCX/PDF bookmark + XLSX sheet-name fixers cataloged `auto` but NOT built; XLSX hidden-content auto-unhide implemented but cataloged human-only.

## index.html data model
All content lives in one `<script>` block:
- `ROWS` ‚Äî 20 SC rows; per row: `a`/`r` = ACP's shipped assessment/remediation tier per format (C/Q/H/NA ¬∑ A/AC/AP/AI/M/N/NA) ‚Äî drives the table's cell colors/labels, the ¬ß2 "Today" chip (assessment), and the ¬ß4 "Current" card (remediation, via `RTIER_HOW`). `ae`/`re` = per-format mechanism detail (`b` argument bullets, `c` concessions, `x` counterargument, `ev` evidence paths, `cat` drift note) ‚Äî bullets + evidence render in drawer ¬ß6 (collapsed "Technical notes"); concessions/counterargument/drift render in ¬ß5 (FAQ) instead.
- `RUBRIC` ‚Äî **the rule-inherent ceiling** ("potential"), independent of ACP's shipped status: per SC, `plain` (2-sentence rule statement) + per-format `{aClass, aWhy, fClass, fWhy}`. `aClass` ‚àà DET/MA/HUM/NAx, `fClass` ‚àà MECH/GEN/REAUTH/NAx ‚Äî both display through the **same three-way vocabulary** now: `RCLASS`/`FCLASS` render DET/MECH‚Üí"Rules-based", MA/GEN‚Üí"AI-assisted", HUM/REAUTH‚Üí"Human-assisted", NAx‚Üí"Not applicable". This is the taxonomy that should NOT churn as ACP ships more. Grounded in `docs/deva-reconciliation.md` (assess ceiling reasoning) and `docs/tier-methodology.md` (fix ceiling reasoning, reframed without the "shipped" qualifier).
- `RTIER_HOW` ‚Äî maps ACP's shipped remediation tier (`ROWS.r[f]`, e.g. A/AC/AP/AI/M/N) onto the *same* DET/MA/HUM/NAx bucket as RUBRIC's `fClass`, so "current" and "potential" read on one shared scale in drawer ¬ß4. Verified per-tier against actual mechanism text, not assumed: AC (1.1.1's vision-conditional auto-apply) ‚Üí AI-assisted; AP (1.3.2 xlsx hidden-content unhide, 2.4.2's filename-derived title) ‚Üí Rules-based, because those specific AP cases are mechanical, not generative. **If a new SC ever uses AC/AP, re-check its actual mechanism before trusting this mapping** ‚Äî it's not a generic rule, it's been verified case-by-case.
- `RANK` + the `.scopenote` in `remediationCVP()` ‚Äî Rules-based/AI-assisted/Human-assisted is **not a strict quality ladder**; it only orders mechanism reliability *within a fixed scope*. Whenever ACP's current fix mechanically solves a narrower slice of a rule than the rule's ceiling covers (e.g. 1.3.2 xlsx: mechanically unhides content, but has no safe fix for merged cells, whose ceiling stays Human-assisted), Current can rank higher than Potential without that being a contradiction ‚Äî it means ACP already nailed the easy sub-case, not that ACP exceeds the ceiling. `RANK` detects this (`RANK[curHow] > RANK[potClass]`) and auto-appends an explanatory note so the badges don't read as self-contradictory. As of this writing it fires on 7 cells: 1.3.2/xlsx, 2.4.6/docx+pptx, 2.4.2/docx+xlsx+pptx+pdf ‚Äî all "presence-only mechanical fix vs. descriptive/generative ceiling" cases. If you add a new SC/format where this pattern could recur, don't hand-flag it ‚Äî the check is generic and will catch it automatically as long as `fClass`/`RTIER_HOW` are set correctly.
- `DEVA` ‚Äî per-SC: Deva's verdict verbatim (`v`, `t`) + our conclusion (`k`: agree|augment|disagree, `ours`, `note`); rendered as an aside under drawer ¬ß3. Also surfaces directly on the table: `DEVA_MARK` (in the table-build loop) tags each ASSESS cell (never remediate ‚Äî her checklist has no remediation column) with a superscript symbol (‚ÄÝaugment / ‚Ä°disagree) **and** a colored corner-flag triangle on the `<td>` itself (`.deva-augment`/`.deva-disagree` CSS classes, purple/red, rendered as a `::after` pseudo-element) ‚Äî two redundant signals so the flag isn't color-only (this tool demos WCAG 1.4.1 Use of Color; it would be a bad look to violate it in its own UI). The corner-flag treatment replaced an earlier full-border-plus-tint version: with ~40% of assess cells marked, a full box around each competed too heavily with the primary tier-color signal (green/blue/orange text) ‚Äî a small corner triangle carries the same information with far less visual weight. The triangle itself is a real CSS `border` trick (equal-width top/left borders set to `transparent`, right border colored), not `box-shadow` ‚Äî box-shadow on `<td>` is unreliable across browsers when the table has `border-collapse:collapse` (the collapse algorithm only resolves actual borders, not shadows); that was tried first and silently failed to render on the live site despite working in local dev preview. The `.legend` reuses the identical `.deva-augment`/`.deva-disagree` classes on a `.swatch` span so the legend visually demonstrates the exact pattern, not just a text description of it. Both only apply **where `RUBRIC[sc].formats[f].aClass !== "NAx"`** ‚Äî a per-SC `k` sometimes covers only some formats (1.3.2's "disagree" is about XLSX/PPTX/PDF beating her class; DOCX is N/A and has no claim to compare, so it gets neither mark even though the row's `k` is "disagree"). If you add a new DEVA entry, don't assume the mark should appear on all 4 assess cells ‚Äî check whether the `note` text's claim actually applies format-by-format first.
- `CHECKLIST2` ‚Äî Deva's SECOND checklist (`Accessibility Checklist.pdf`, the manual per-file SOP), keyed `sc ‚Üí format ‚Üí [{c: criteria name, h: how-to-check-or-fix text}]` ‚Äî inherently per-format (unlike `DEVA`, which is SC-level), since each checklist page only covers one format. Only the 28 items verified as an actual match (üü¢/üü° in `docs/deva-checklist-reconciliation.md`) are present; the 10 confirmed gaps and 6 not-actually-a-WCAG-SC items are deliberately absent ‚Äî don't add them here even for completeness, since showing a gap as "today's manual step" would misrepresent it. Renders in drawer ¬ß4 as a "HITL today" block (`.hitl` CSS classes), directly under the CVP cards and `fWhy` ‚Äî grounds the Human-assisted/Current mechanism in Deva's team's actual procedure rather than an abstract label. Some cells combine multiple checklist rows into one array (e.g. DOCX 1.3.1 holds both "Uses Heading Styles" and "Simple Tables with Headers," since both map to the same SC√óformat) ‚Äî check for this before assuming one entry per cell.
- `W3SLUG` ‚Äî per-SC anchor into https://www.w3.org/TR/WCAG22/ (canonical), W3C Understanding, and W3C Techniques (quickref) links ‚Äî all three shown in drawer.
- `PRINCIPLE` ‚Äî per-SC WCAG Principle‚ÜíGuideline breadcrumb text (factual hierarchy, not the SC text itself) ‚Äî shown under the h2.
- `QUOTE` ‚Äî per-SC short (<15-word) excerpt of the SC's own normative wording, quotation-marked and attributed with a link back to the canonical text ‚Äî NOT the full SC text (deliberately trimmed; don't expand these into full reproductions).
- `UPGRADE` ‚Äî per-SC, one hand-authored sentence: the cheapest concrete build that would close the gap between current and potential. Shown directly in drawer ¬ß4 ("Gap to close") ‚Äî not in the FAQ (moved there to avoid duplicating the same sentence twice in one drawer).

Drawer render: `renderDrawer(sc, f, trigger)` (called from both the table's click handler and cross-link clicks) builds a **unified 6-section drawer per (SC, format)**:
1. The rule, in plain words
2. Can this be assessed automatically? ‚Äî RUBRIC ceiling badge + an inline "Today: ‚Ä¶" chip (`assessTodayChip()`, from `ROWS.a[f]`) ‚Äî assessment does NOT get the current/potential merge that remediation does; it's one badge with a status chip beside it.
3. Why it lands there [+ the Deva aside, if this SC has one]
4. **Remediation ‚Äî current vs. potential** (`remediationCVP()`) ‚Äî a side-by-side comparison card, not two separate sections: Current (ACP today, from `ROWS.r[f]` via `RTIER_HOW`) vs. Potential (RUBRIC's `fClass`), both labeled on the Rules-based/AI-assisted/Human-assisted scale, followed by `fWhy` and the `UPGRADE` "Gap to close" note. This replaced the old split of "¬ß4 ceiling badge" + "¬ß5 ACP-today status line" ‚Äî don't reintroduce that split without a reason; the merge was a deliberate scope decision (assessment kept its old two-part structure, remediation didn't).
5. FAQ (`buildFAQ()`) ‚Äî mostly derived from data already on `ROWS.ae`/`ROWS.re` (first argument bullet, concessions, counterargument, catalog drift), so it doesn't require separate per-format authoring. Does NOT repeat the "gap to close" content ‚Äî that lives only in ¬ß4 now.
6. Technical notes (collapsed `<details>`) ‚Äî mechanism bullets + evidence only; concessions/counterarguments/drift live in the FAQ instead, not duplicated here.

`linkify()` auto-links bare SC-code mentions (e.g. "1.3.2") inside plain/why/FAQ/gap text to that SC's own drawer ‚Äî don't hand-author cross-link markup, it's automatic from any text that mentions another SC's code. Ceiling tiles above the table (`#ceilingTiles`, `#acpTiles`) are computed at load time by iterating `ROWS`√ó`RUBRIC` ‚Äî never hand-count these; the IIFE right after the table-build loop is the single source.

**Accessibility of the tool itself** (dogfooded, since it's a WCAG demo): table cells are real `<button>` elements (not click-handler `<td>`s) with descriptive `aria-label`s ‚Äî natively keyboard-reachable. The drawer (`#panel`) has `role="dialog" aria-modal="true" aria-labelledby="panel-title"`, a focus trap (Tab cycles within it), Escape-to-close, and focus returns to the triggering button on close (`lastTrigger` tracks this ‚Äî always pass `trigger` to `renderDrawer()`, including from cross-link clicks, or focus-return breaks). If you add new interactive elements to the drawer template, they need `:focus-visible` styling (already covers `button`/`a`/`.xlink` generically) and must be reachable within the existing focus-trap query (`button, a[href], [tabindex="0"]`).

Validate any index.html edit: extract the script block and eval ROWS/RUBRIC/DEVA/W3SLUG/PRINCIPLE/QUOTE/UPGRADE/RTIER_HOW ‚Äî every SC in ROWS must have a RUBRIC entry with all 4 formats populated, NA-consistency should hold (aClass NAx ‚áí fClass NAx), `buildFAQ()` should return ‚â•1 item for every SC√óformat, and `remediationCVP()`/`assessTodayChip()` should never render the string "undefined". A quick in-browser integrity check (paste into devtools or `javascript_tool`):
```js
for (const row of ROWS) { const r = RUBRIC[row.sc]; if (!r) console.error(row.sc, 'missing'); for (const f of ["docx","xlsx","pptx","pdf"]) { const e = r.formats[f]; if (!e || !e.aClass || !e.fClass) console.error(row.sc, f, 'incomplete'); if (buildFAQ(row, f, e).length === 0) console.error(row.sc, f, 'no FAQ'); if (remediationCVP(row, f, e).includes('undefined')) console.error(row.sc, f, 'cvp broken'); } }
```

## Mobile / responsive
- `#panel` goes full-width (`width:100%`, no left border) below 640px; `.cvp` (the current-vs-potential cards in ¬ß4) stacks vertically below the same breakpoint ‚Äî both via one `@media (max-width:640px)` block. `#panel{overflow-x:hidden}` is a defensive second rule (deliberately separate, later in source order, so it overrides only the `overflow-x` half of the base rule's `overflow:auto` shorthand) ‚Äî keep it if you ever add wide content (tables, code blocks) to the drawer.
- `#backdrop` (a fixed, semi-transparent overlay, click-to-close) plus `document.querySelector(".wrap").inert = true/false` toggle in `renderDrawer()`/`closePanel()` give the dialog real modal isolation ‚Äî the background table is genuinely unclickable and untabbable while the drawer is open, not just visually dimmed. `aria-modal="true"` alone does NOT enforce this in browsers; `inert` is what actually does.
- **Known tooling gap, not a page bug:** this repo's dev-session browser-preview tool has a reproducible viewport desync at narrow widths ‚Äî `window.innerWidth` reports a stale wide value (differs from `outerWidth`/`screen.width`) after `resize_window` to a mobile preset, and screenshots taken in that state show clipped/scrolled-looking text even though every computed style and `scrollLeft` checks out correct. Verify mobile CSS via `getComputedStyle()`/`matchMedia()` assertions (as done for this change), not by trusting narrow-viewport screenshot pixels in this environment ‚Äî cross-check against a real device or a different browser tool if a pixel-level check is ever needed.
